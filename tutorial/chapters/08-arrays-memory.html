Chapter 8: Arrays and Memory — Managing Your Own Resources

Arrays: Rows of Mailboxes

An array is a contiguous block of memory holding multiple values of the same type. Think of it as a row of identical mailboxes, numbered starting from zero. You declare one by putting the count in square brackets: 

int scores[5];          // 5 integers, side by side in memory
scores[0] = 90;         // first element (index 0)
scores[1] = 85;         // second element
scores[4] = 72;         // last element (index 4, NOT 5!)

The most important thing to remember: array indices start at 0 and the last valid index is size - 1. An array of 5 elements has indices 0, 1, 2, 3, 4. There is no index 5. If you access scores[5], you're reading or writing memory that doesn't belong to your array. C won't stop you. That's your problem now.

You can also initialize an array on one line: 

int scores[5] = { 90, 85, 78, 92, 72 };

Crystal's Window Array

Crystal tracks every window on your screen in a fixed-size array: 

struct WindowTexture windows[MAX_CLIENTS];   // MAX_CLIENTS = 256
int window_count;                             // how many are actually used

This is a classic pattern in C: allocate a fixed-size array and keep a separate count of how many slots are actually filled. When a new window appears, Crystal puts it in windows[window_count] and increments the count. When a window closes, it removes it from the array and decrements the count.

The window manager does the same thing: 

Client clients[MAX_CLIENTS];   // the window list
int num_clients;               // how many are active

When it needs to find a particular window, it loops through: 

for (int i = 0; i < wm->num_clients; i++) {
    if (wm->clients[i].client == w)
        return &wm->clients[i];
}

This is linear search — check each element one by one. For 256 elements, it's plenty fast. For a million elements, you'd want a smarter data structure. But for a window manager? Nobody has 256 windows open. (And if they do, they need a vacation, not a faster algorithm.)

Stack vs. Heap: Where Memory Lives

Every variable you've seen so far lives on the stack. The stack is automatic memory — the compiler handles it for you. When you declare int x = 5; inside a function, space for x is carved out on the stack when the function is called and reclaimed when the function returns. You don't have to do anything.

But the stack has limits. It's small (typically 1-8 MB), and everything on it disappears when the function returns. What if you need memory that:

  • Survives after the function that created it returns?
  • Is too large for the stack (think: a 4K image buffer)?
  • Needs to have its size determined at runtime?

That's where the heap comes in. The heap is manual memory — you ask for it explicitly, and you must give it back explicitly. The tools are malloc, calloc, and free.

malloc and free: You Asked for It, You Return It

#include <stdlib.h>   // for malloc, calloc, free

// Allocate space for 100 integers on the heap
int *numbers = malloc(100 * sizeof(int));

// Use it just like a regular array
numbers[0] = 42;
numbers[99] = 17;

// When you're done, give the memory back
free(numbers);

malloc(size) asks the operating system for size bytes of memory and returns a pointer to it. sizeof(int) tells you how many bytes an int takes (usually 4), so 100 * sizeof(int) is 400 bytes. The memory is yours until you call free().

calloc is like malloc but with two differences: it takes two arguments (count and size per element) and it zero-initializes the memory. malloc gives you whatever garbage was there before. calloc gives you clean zeros.

Crystal's Shadow Buffer: calloc in the Wild

Here's the real code where Crystal allocates memory for shadow computation. This is from the Gaussian blur routine that generates the soft shadows behind every window: 

// Allocate a single-channel (grayscale) buffer for the shadow.
// calloc zero-initializes, so the shadow starts fully transparent.
unsigned char *shadow = calloc((size_t)shadow_w * shadow_h, 1);
if (!shadow) return;

// ... hundreds of lines of blur math ...

// Temporary buffer for ping-ponging between blur passes
unsigned char *temp = calloc((size_t)shadow_w * shadow_h, 1);
if (!temp) { free(shadow); return; }

// ... do all three blur passes ...

// Upload the blurred result to the GPU, then free the CPU buffers
free(temp);
free(shadow);

A few things to notice here:

  • calloc(shadow_w * shadow_h, 1) allocates enough bytes for a grayscale image of width shadow_w and height shadow_h. Each pixel is 1 byte.
  • After calloc, the code checks if the pointer is NULL. malloc and calloc return NULL if the system is out of memory. Always check.
  • When the second calloc fails, the code frees the first buffer before returning. If it didn't, that memory would leak.
  • Both buffers are freed at the end. The shadow data has been uploaded to the GPU, so the CPU copies are no longer needed.

This is the rhythm of manual memory management: allocate, use, free. Every malloc or calloc must eventually have a matching free. It's like checking out a library book — you have to return it, or eventually the library runs out of books.

Memory Leaks

A memory leak happens when you allocate memory but never free it. Your program slowly eats more and more RAM until it either runs out or the system kills it. The terrifying thing is that the program keeps working — it just gets hungrier over time, like a tapeworm. 

// Memory leak! The pointer is lost, memory is never freed.
void leak_example(void)
{
    int *data = malloc(1000 * sizeof(int));
    // ... use data ...
    return;   // oops — never called free(data)!
}

Every time this function is called, 4000 bytes of memory disappear forever (until the program exits). Call it a million times and you've leaked 4 GB. In a long-running program like a compositor that runs for hours or days, memory leaks are critical bugs.

A tool called Valgrind can track every allocation in your program and tell you exactly which ones were never freed. It's like an X-ray for memory bugs. Run your program through Valgrind during development and it will save you from the most insidious class of C bugs.

Buffer Overflows

Writing past the end of an array is called a buffer overflow. It's one of the most dangerous bugs in computing — not just a crash, but a potential security vulnerability. 

char name[8];
strcpy(name, "Alexander");   // "Alexander" is 9 chars + null = 10 bytes
                              // but name only has room for 8!

The string "Alexander" is too long for the buffer. The extra characters overflow into adjacent memory, corrupting whatever was stored there. If an attacker controls the data, they can potentially overwrite return addresses on the stack and hijack the program's execution. This is how many real-world exploits work — buffer overflows in C programs have been responsible for some of the biggest security breaches in history.

The fix? Always check sizes. Use strncpy instead of strcpy. Use snprintf instead of sprintf. Never trust user input to be the right length. Crystal's code uses snprintf everywhere it builds strings: 

// Safe — snprintf will never write more than sizeof(path) bytes
snprintf(path, sizeof(path),
         "%s/.local/share/aqua-widgets/sl_close_button.png", home);

Arrays and Pointers: The Connection

Here's a C secret that confuses everyone at first: arrays and pointers are deeply related. When you pass an array to a function, it "decays" into a pointer to its first element. The function receives a pointer, not a copy of the entire array. 

void print_array(int *arr, int len)
{
    for (int i = 0; i < len; i++) {
        printf("%d ", arr[i]);
    }
    printf("\n");
}

int numbers[] = { 10, 20, 30, 40 };
print_array(numbers, 4);   // passes a pointer, not a copy

That's why functions that take arrays always need the length as a separate argument — the pointer doesn't carry size information. You have to track it yourself. This is another spot where C gives you power at the cost of responsibility.

Try It Yourself

  1. Allocate an array of 100 integers with malloc. Fill it with squares: arr[0] = 0, arr[1] = 1, arr[2] = 4, arr[3] = 9... Print the first 10 values, then free the array.
  2. Write a function int sum(int *arr, int len) that returns the sum of an array. Test it with a few different arrays.
  3. Challenge: Write a function that takes two arrays and their lengths, allocates a new array big enough to hold both, copies both into it (use a loop), and returns the new array. The caller is responsible for freeing it. This is a simple version of what Crystal does when it reorders its window array.
