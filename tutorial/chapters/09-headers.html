Chapter 9: Header Files — Sharing Your Code's Menu

The Problem

Your program is growing. You've got functions in one file that need to call functions in another file. But C compiles each .c file separately. When the compiler is working on main.c, it has no idea what functions exist in crystal.c. If you write crystal_init(&wm) in main.c, the compiler says: "I have no idea what crystal_init is. I've never heard of it. Refuse to compile."

Fair enough. The compiler needs to know what the function looks like — its name, its parameters, its return type — before it can call it. That's where header files come in.

Headers: The Menu, Not the Kitchen

A header file (.h) is like a restaurant menu. It tells you what's available and what each item looks like, but it doesn't contain the recipes. The recipes live in the .c files. 

// crystal.h — the "menu"
bool crystal_init(AuraWM *wm);
void crystal_shutdown(AuraWM *wm);
void crystal_composite(AuraWM *wm);
// crystal.c — the "kitchen"
bool crystal_init(AuraWM *wm)
{
    // 200+ lines of actual initialization code
    // ...
}

When main.c says #include "crystal.h", the preprocessor literally copies the contents of crystal.h into main.c before compilation. Now the compiler knows what crystal_init looks like and can verify you're calling it correctly. The actual code (the "recipe") gets linked in later, during the linking step.

crystal.h: A Real Header

Here's the structure of Crystal Compositor's actual header file: 

#ifndef AURA_CRYSTAL_H
#define AURA_CRYSTAL_H

#include "wm.h"
#include <stdbool.h>

// Shadow parameters
#define SHADOW_RADIUS       22
#define SHADOW_Y_OFFSET      4
#define SHADOW_ALPHA_ACTIVE  0.45

// Public functions — the things other files can call
bool crystal_init(AuraWM *wm);
void crystal_shutdown(AuraWM *wm);
void crystal_composite(AuraWM *wm);
void crystal_window_mapped(AuraWM *wm, Client *c);
void crystal_window_unmapped(AuraWM *wm, Client *c);
void crystal_window_damaged(AuraWM *wm, Client *c);
bool crystal_is_active(void);

#endif // AURA_CRYSTAL_H

Let's break down the parts:

Include Guards

#ifndef AURA_CRYSTAL_H
#define AURA_CRYSTAL_H
// ... everything in between ...
#endif

These three lines are an include guard. They prevent the header from being included twice in the same file. Here's why that matters: imagine events.c includes both crystal.h and wm.h, and crystal.h also includes wm.h. Without the guard, wm.h gets pasted in twice, causing "duplicate definition" errors. The guard says: "If AURA_CRYSTAL_H hasn't been defined yet, define it and include everything. If it's already defined, skip the whole file." Problem solved.

Every header file you write should have an include guard. The convention is to use the filename in all caps with underscores: MY_FILE_H.

Angle Brackets vs. Quotes

#include <stdbool.h>    // system header — search system directories
#include "wm.h"          // project header — search current directory first

Angle brackets (< >) tell the preprocessor to look in the system include directories — that's where standard library headers and installed library headers live. Double quotes (" ") tell it to look in the current directory first, then fall back to system directories. Use quotes for your own headers, angle brackets for everything else.

What Goes in a Header

  • Function declarations (prototypes) — the name, parameters, and return type.
  • Struct definitions — so other files can use your types.
  • #define constants — like SHADOW_RADIUS.
  • Type aliases — typedef declarations.

What does NOT go in a header: function bodies (implementations), variable definitions (unless extern), or anything that produces actual code. Headers are declarations. Source files are definitions.

How Files Talk to Each Other

The AuraOS window manager has about a dozen source files. Here's how they connect: 

// main.c includes headers for every module it uses
#include "wm.h"
#include "events.h"
#include "crystal.h"

// Now main.c can call functions from any of those modules
wm_init(&wm);
crystal_init(&wm);
events_run(&wm);

Each .c file is its own world. It doesn't know about any other file unless it explicitly includes that file's header. This is how C programs stay organized as they grow — each module has a clean public interface (the header) and a private implementation (the source file). Everything that's static in a .c file is completely invisible to other files.

This is C's version of encapsulation. Languages like Java have public and private keywords. C has a simpler system: if it's in the header, it's public. If it's static in the source file, it's private. Crystal has dozens of static helper functions that only crystal.c can see — like static void generate_shadow_texture(). That's internal-only code that other modules have no business calling.

Try It Yourself

  1. Create a simple math library: mymath.h with declarations for int add(int a, int b) and int multiply(int a, int b). Then mymath.c with the implementations. Then main.c that includes mymath.h and uses both functions.
  2. Compile them together: gcc main.c mymath.c -o mathtest
  3. Add include guards to mymath.h. Try including the header twice in main.c — the guard should prevent duplicate definition errors.
