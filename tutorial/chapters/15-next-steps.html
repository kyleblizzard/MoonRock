Chapter 15: Next Steps

Where to go from here — you're not a beginner anymore

You've made it. Fourteen chapters of C, from "what is a variable" to building algorithms that blur pixels on a GPU. You've seen real code from a real compositor that renders a real desktop. That's not nothing. That's actually a lot.

So where do you go from here? The honest answer is: wherever you want. But here are the paths I'd recommend, in order of "things that will make you dangerous."

Read These Books

You already own two of the best C books ever written. Now's the time to actually read them cover to cover:

  • "The C Programming Language" by Kernighan & Ritchie — The original. Written by the people who invented C. It's short, precise, and every sentence matters. Read chapters 1-6 carefully. Do the exercises. When you finish this book, you'll understand C better than most professional programmers.
  • "Absolute Beginner's Guide to C" by Greg Perry — You've already gone past "absolute beginner," but go back and read the chapters on file I/O, string manipulation, and the standard library. These are the practical tools you'll use every day.

Take These Free Courses

  • CS50 (Harvard) — The most popular intro CS course in the world. Free. Starts with Scratch, moves to C, then Python. The C weeks are gold. David Malan is an incredible teacher.
  • Beej's Guide to C Programming — Free online book. More modern than K&R, covers C11/C17 features. Great reference when you forget how something works.
  • learn-c.org — Interactive C tutorials in your browser. Good for practice when you don't want to set up a compiler.

Watch These People

  • Casey Muratori — Handmade Hero (handmadehero.org) — A professional game developer builds an entire game engine from scratch in C, on camera, explaining everything. Hundreds of episodes. This is the closest thing to an apprenticeship you'll find for free. Warning: you will learn so much it might ruin you for frameworks forever.
  • Tsoding Daily (YouTube) — Entertaining, fast-paced C and systems programming. Builds things from scratch. Good energy. You'll learn things you didn't know you needed to know.
  • Ben Eater (YouTube) — Builds a computer from logic gates. Doesn't teach C directly, but understanding what the hardware is actually doing beneath your code changes how you think about programming forever.

Build Something

The single most important thing you can do now is build something you care about. Not a tutorial project. Not someone else's exercise. Something YOU want to exist.

Ideas, in order of difficulty:

  1. A text file search tool — like a mini grep. Read files, search for patterns, print matches.
  2. A simple HTTP server — serve static files over TCP sockets. You'll learn networking and the HTTP protocol.
  3. A terminal text editor — handle keyboard input, render text to the terminal, implement basic editing. Start with kilo as a reference.
  4. A chip-8 emulator — emulate a simple CPU. You'll learn bitwise operations, memory mapping, and how computers actually execute instructions.
  5. Contribute to AuraOS — you've been reading Crystal Compositor's code for 15 chapters. You understand the structs, the event loop, the shader pipeline. Open an issue. Fix a bug. Add a feature. The codebase is yours to explore.

Tools to Install

  • valgrind — finds memory leaks and buffer overflows. Run your programs through it regularly.
  • gdb — the GNU debugger. Learn the basics: break, run, print, backtrace. It will save you hours.
  • AddressSanitizer — compile with -fsanitize=address and it catches memory errors at runtime. Easier than valgrind for quick checks.
  • Compiler Explorer — paste your C code, see the assembly output. Understanding what the compiler does with your code is a superpower.

Reference Sites

  • cppreference.com/w/c — the definitive C standard library reference
  • cdecl.org — paste a confusing C declaration, get a plain English explanation
  • man7.org — Linux man pages online (system calls, library functions)

One Last Thing

C is not a language you "learn" and then you're done. It's a language you grow into. Every year you write C, you'll understand it a little deeper. You'll write fewer bugs. You'll see patterns you didn't see before. You'll read code that seemed impossible six months ago and think "oh, that's actually elegant."

The Crystal Compositor you've been studying throughout this tutorial is 11,000+ lines of C. It manages OpenGL contexts, X11 windows, shader programs, animation state machines, touch input, display rotation, and plugin systems. Every line was written by someone who started exactly where you are now — staring at a blinking cursor, wondering what #include <stdio.h> means.

Now you know. Go build something.
