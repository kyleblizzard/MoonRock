Chapter 4: Functions — Teaching Your Program New Tricks

One Thing at a Time

Imagine you're giving someone directions to your house. You could write it as one massive paragraph: "Go north on Main, turn left at the gas station, pass the school, turn right at the stop sign, go up the hill..." But it's way easier to break it into steps: "Get on Main Street. Head to the gas station. Turn left. Look for the school..."

Functions are exactly that: named steps. You take a chunk of code that does one specific thing, give it a name, and then you can call it whenever you need that thing done. Instead of copying and pasting the same ten lines everywhere, you write them once and call the function by name. It keeps your code organized, readable, and way less error-prone.

Anatomy of a Function

A function has four parts: a return type, a name, parameters, and a body. 

int add(int a, int b)
{
    int result = a + b;
    return result;
}

Let's break that down:

  • int — the return type. This function gives back a whole number when it's done.
  • add — the name. What we call this function. Should describe what it does.
  • (int a, int b) — the parameters. The inputs. This function needs two integers to work with.
  • { ... } — the body. The actual instructions. Add the two numbers, return the result.

To use (or "call") this function: 

int total = add(3, 7);
printf("Total: %d\n", total);   // Total: 10

When you write add(3, 7), C jumps into the add function, sets a to 3 and b to 7, runs the body, and comes back with the result. It's like calling someone on the phone: "Hey, what's 3 plus 7?" "10." "Thanks." Done.

Void Functions: No Return Value

Not every function needs to give something back. Sometimes you just want it to do something — print a message, update a variable, clean up resources. For those, you use void as the return type: 

void greet(char *name)
{
    printf("Hello, %s!\n", name);
}

// Calling it:
greet("Kyle");   // prints: Hello, Kyle!

void means "nothing." The function doesn't return a value — it just does its job and comes back. You don't write return in a void function (though you can write return; by itself to exit early).

Crystal's Functions: The Real Deal

Now let's look at a real function signature from Crystal Compositor. Here's the initialization function: 

bool crystal_init(AuraWM *wm);

This is a function declaration (also called a "prototype"). It tells the compiler: "there exists a function called crystal_init that takes one parameter and returns a bool." Let's decode each piece:

  • bool — the return type. This function returns true or false. True means "initialization succeeded." False means "something went wrong, we can't composite."
  • crystal_init — the name. Clear and descriptive: it initializes the Crystal compositor.
  • AuraWM *wm — the parameter. It takes a pointer to an AuraWM struct. Don't worry about pointers or structs yet — we'll cover both soon. For now, just think of it as "it needs access to the window manager's state to do its job."

And here's how it's called in main(): 

if (!crystal_init(&wm)) {
    fprintf(stderr, "[aura-wm] Crystal unavailable — no shadows\n");
}

That ! is the "not" operator. So this reads as: "if crystal_init did NOT succeed, print an error message." The program doesn't crash — it just runs without shadows. That's graceful degradation, and it's good design. Your program should handle failure, not ignore it.

Now look at the shutdown function: 

void crystal_shutdown(AuraWM *wm);

void return type — it doesn't return a value. It just cleans up: destroys textures, frees memory, shuts down OpenGL. When you're tearing down, there's nothing meaningful to return. Either it cleaned up or the program is exiting anyway.

Why Functions Matter

Look at Crystal's main.c again. The entire program is basically: 

wm_init(&wm);        // Set up the window manager
crystal_init(&wm);   // Set up the compositor
struts_init(&wm);    // Set up screen edge reservation
decor_init(&wm);     // Load window decoration assets
input_setup_grabs(&wm);  // Set up keyboard shortcuts
events_run(&wm);     // Run forever, processing events
crystal_shutdown(&wm);   // Tear down compositor
wm_shutdown(&wm);    // Tear down window manager

Each of those function calls hides hundreds of lines of code behind a clear, descriptive name. You can read main() like a story: initialize, run, shut down. You don't need to see the details of how each step works to understand the big picture. That's the magic of functions — they let you think at different levels of detail.

crystal_init alone is over 200 lines — it checks for OpenGL extensions, creates rendering contexts, sets up damage tracking, and computes a Gaussian blur kernel. But from main's perspective? It's one line. One function call. "Initialize Crystal." Done.

Writing Your Own Functions

Let's write a complete program with functions. Here's a temperature converter: 

#include <stdio.h>

// Convert Fahrenheit to Celsius
float to_celsius(float fahrenheit)
{
    return (fahrenheit - 32.0) * 5.0 / 9.0;
}

// Convert Celsius to Fahrenheit
float to_fahrenheit(float celsius)
{
    return celsius * 9.0 / 5.0 + 32.0;
}

// Print a temperature in both units
void print_temp(float f)
{
    printf("%.1f°F = %.1f°C\n", f, to_celsius(f));
}

int main(void)
{
    print_temp(32.0);    // freezing point
    print_temp(72.0);    // room temperature
    print_temp(212.0);   // boiling point
    return 0;
}

Notice how print_temp calls to_celsius — functions can call other functions. In fact, that's how all programs work. Functions calling functions calling functions, all the way down to the ones that actually talk to the hardware. It's functions all the way down.

Naming matters. Notice how Crystal names its functions: crystal_init, crystal_shutdown, crystal_composite, crystal_window_mapped. They all start with crystal_ so you know which module they belong to, followed by a verb that describes the action. This naming convention is how C programmers organize code without classes or namespaces. Keep your function names clear and consistent.
