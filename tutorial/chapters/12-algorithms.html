Chapter 12: Algorithms — The Shadow Behind Every Window

What Is an Algorithm?

An algorithm is just a step-by-step procedure for solving a problem. A recipe is an algorithm. Driving directions are an algorithm. "Check each sock in the pile until you find a matching pair" is an algorithm (a linear search, to be precise).

We've already seen simple algorithms — linear search through the client array, iterating over windows. But Crystal has a genuinely interesting algorithm hiding inside it: the 3-pass box blur approximation of Gaussian blur. It's how Crystal generates those soft, realistic shadows behind every window. Let's tear it apart.

Why Shadows Need Blur

A shadow isn't just a dark rectangle behind a window. Real shadows are soft — they're darkest near the object and gradually fade to nothing at the edges. That soft gradient is created by blurring a hard-edged shape.

The gold standard for smooth blurring is Gaussian blur, named after the bell curve. A true Gaussian blur looks at every pixel in a large radius around each output pixel, weighing nearby pixels heavily and distant pixels lightly. The result is a beautifully smooth fade. The problem? A true Gaussian blur is expensive. For a blur radius of 22 pixels, each output pixel needs to sample a 45x45 grid of neighbors. That's over 2,000 lookups per pixel. For a 1920x1080 shadow, that's billions of operations per frame. Way too slow.

The Trick: Box Blur x3

Here's the clever part. A box blur (a simple averaging filter) applied three times in succession closely approximates a Gaussian blur. This isn't a hack — it's mathematically proven. And a box blur can be computed in O(1) per pixel using a sliding window, regardless of the blur radius. That changes everything.

Crystal's algorithm works in five steps:

Step 1: Create the Shadow Shape

// Allocate a grayscale buffer — one byte per pixel
unsigned char *shadow = calloc((size_t)shadow_w * shadow_h, 1);

// Fill in a solid white rectangle where the window sits
for (int y = pad; y < pad + chrome_h; y++) {
    memset(&shadow[y * shadow_w + pad], 255, (size_t)chrome_w);
}

We start with a blank buffer (all zeros — fully transparent) and paint a solid white rectangle where the window is. This is the "seed" — the hard-edged shape that we'll blur into a soft shadow. The pad around the rectangle gives the blur room to fade out.

Step 2: Horizontal Box Blur

For each row of pixels, slide a window of width kernel_width across the row. Keep a running sum as you go. Each output pixel is the average of its horizontal neighbors. 

for (int y = 0; y < shadow_h; y++) {
    int sum = 0;
    // Build initial sum for the first window position
    for (int k = -blur_radius; k <= blur_radius; k++) {
        int sx = k < 0 ? 0 : k;   // clamp to edge
        sum += shadow[y * shadow_w + sx];
    }
    temp[y * shadow_w] = sum / kernel_width;

    // Slide the window across the row: add the new pixel, drop the old one
    for (int x = 1; x < shadow_w; x++) {
        // Add the pixel entering the window on the right
        // Subtract the pixel leaving the window on the left
        // Divide by kernel_width to get the new average
        temp[y * shadow_w + x] = new_sum / kernel_width;
    }
}

The key insight: when the window slides one pixel to the right, you don't recompute the entire sum. You just add the new pixel entering on the right and subtract the pixel leaving on the left. That's two operations per pixel, no matter how large the blur radius is. That's what O(1) per pixel means.

Step 3: Vertical Box Blur

Same thing, but sliding vertically instead of horizontally. This makes the blur "separable" — a 2D blur decomposed into two 1D blurs. A 2D box blur over a 45x45 area would need 2,025 lookups per pixel. Two 1D blurs of width 45 need only 90 lookups total (and with the sliding window trick, just 4 operations).

Step 4: Repeat Two More Times

Crystal runs the horizontal+vertical pass three times. Each pass smooths the result further. After one pass, the shadow has blocky, square edges. After two passes, it's smoother. After three passes, it's virtually indistinguishable from a true Gaussian blur.

Step 5: Upload to GPU

The final blurred buffer is uploaded to the GPU as an OpenGL texture. Crystal draws it as a semi-transparent black quad behind the window. The alpha channel comes from the blur buffer — bright pixels are opaque shadow, dark pixels are transparent. The result: a soft, realistic drop shadow that fades smoothly from the window's edge.

And here's the beautiful part: the shadow texture is cached. It's only regenerated when the window resizes. During normal operation — moving windows, switching focus, scrolling — the shadow is just a pre-computed texture. Zero CPU cost.

Why This Matters

This algorithm is a perfect example of a pattern you'll see everywhere in programming: take an expensive operation, find a mathematically equivalent but cheaper approximation, and cache the result. Crystal's shadows look identical to a true Gaussian blur but compute in a fraction of the time.

The same box-blur-x3 trick is used in CSS blur() filters, Photoshop's Gaussian blur, iOS's frosted glass effects, and dozens of other places. It's one of those algorithms that appears simple on the surface but has deep mathematical elegance underneath.

The math behind it: The central limit theorem tells us that convolving any distribution with itself enough times approaches a Gaussian (bell curve). A box function convolved with itself three times gives a very good Gaussian approximation. This was first described in W. Wells' 1986 paper and has been a standard trick in graphics ever since.

Try It Yourself

  1. Write a program that creates an array of 20 integers, fills the middle 10 with the value 255 (and the rest with 0), then applies a simple box blur with radius 2. Print the array before and after the blur to see the smoothing effect.
  2. Modify your blur to run 3 passes. Compare the output to a single pass — notice how three passes produce a smoother, more gradual falloff.
  3. Challenge: Implement the sliding window optimization. Instead of re-summing all values in the kernel for each position, add the new value and subtract the old one. Verify your output matches the naive version.
