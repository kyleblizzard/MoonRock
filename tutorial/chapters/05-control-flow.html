Chapter 5: Making Decisions — if, else, loops

Programs That Think

Up until now, our programs have been railroads — one track, no turns, start to finish. That's fine for hello world, but real programs need to make decisions. "If the user clicked the close button, close the window. If they dragged the title bar, move the window. If neither, do nothing." Without branching and looping, a program can only do exactly one thing in exactly one way. That's not a program — that's a recipe card.

Control flow is what makes programs interesting. It's the difference between a player piano that plays one song and a musician who responds to the audience.

if / else: Making Decisions

The simplest form of decision-making: 

if (temperature > 100) {
    printf("It's boiling!\n");
}

If the condition inside the parentheses is true, the code inside the braces runs. If it's false, we skip it entirely. Simple as that.

You can add an else for the alternative: 

if (temperature > 100) {
    printf("It's boiling!\n");
} else {
    printf("Not boiling yet.\n");
}

And you can chain them for multiple options: 

if (temperature > 100) {
    printf("Boiling!\n");
} else if (temperature > 50) {
    printf("Hot.\n");
} else if (temperature > 20) {
    printf("Comfortable.\n");
} else {
    printf("Cold.\n");
}

Only one of these branches will run — C checks them top to bottom and takes the first one that's true. If none are true, it runs the else (if there is one).

Comparison Operators

The things you put inside if conditions: 

a == b    // equal to (two equals signs! one = is assignment)
a != b    // not equal to
a > b     // greater than
a < b     // less than
a >= b    // greater than or equal
a <= b    // less than or equal
The Classic Mistake: Writing = when you mean ==. if (x = 5) doesn't check if x equals 5 — it assigns 5 to x, which is always true. This bug has bitten every C programmer alive. The compiler might warn you, but it's perfectly legal C. Watch for it.

Crystal's Error Checking: if in the Wild

Here's a pattern you'll see constantly in Crystal's code — checking if something failed and handling the error: 

// From crystal_init() — checking if GLX is available
int glx_error_base, glx_event_base;
if (!glXQueryExtension(wm->dpy, &glx_error_base, &glx_event_base)) {
    fprintf(stderr, "[crystal] ERROR: GLX extension not available.\n");
    return false;
}

glXQueryExtension() returns true if GLX is available, false if it's not. The ! flips it: "if NOT available, print an error and bail out." This is the bread and butter of C programming — try something, check if it worked, handle the failure. You'll write this pattern hundreds of times.

And here's how the main function uses it: 

if (!wm_init(&wm, display)) {
    return 1;   // couldn't initialize — exit with error code
}

if (!crystal_init(&wm)) {
    fprintf(stderr, "[aura-wm] Crystal unavailable — no shadows\n");
    // Note: we DON'T return here! The WM continues without compositing.
}

Notice the difference: if the window manager can't initialize, the program exits (returns 1). But if Crystal can't initialize, the program keeps running — you just don't get fancy shadows. The if statement lets you make these kinds of nuanced decisions about how to handle different situations.

while Loops: Keep Going

A while loop repeats a block of code as long as a condition is true: 

int count = 1;
while (count <= 5) {
    printf("%d\n", count);
    count++;   // count++ is shorthand for count = count + 1
}

This prints 1 through 5. Each time through the loop, it checks: is count still less than or equal to 5? If yes, run the body again. If no, stop.

The ++ operator increments a variable by 1. It's so common in C that the C++ language was literally named after it. (Get it? C, plus plus. C++. I'll see myself out.)

Crystal's Event Loop: The Heartbeat

Here's the most important while loop in the entire window manager. This is the event loop from events.c — the heartbeat of the compositor: 

while (wm->running) {
    XEvent ev;
    XNextEvent(wm->dpy, &ev);

    // Let Crystal handle compositor events first
    if (crystal_handle_event(wm, &ev)) {
        // Crystal consumed the event — skip normal dispatch
    } else if (ev.type < LASTEvent && handlers[ev.type]) {
        handlers[ev.type](wm, &ev);
    }

    // Draw all windows with shadows
    if (crystal_is_active()) {
        crystal_composite(wm);
    }
}

This loop runs forever — or rather, until someone sets wm->running to false (which happens when you press Ctrl+C or log out). Each time through, it:

  1. Waits for the next event from the X server (a mouse click, a key press, a window resize...).
  2. Lets Crystal handle it if it's a compositor event (like damage).
  3. Otherwise, dispatches it to the appropriate handler function.
  4. Redraws all the windows on screen.

This is the core of every interactive program you've ever used. Your browser has an event loop. Your phone's OS has an event loop. Every video game has an event loop. It's always the same pattern: wait for input, process it, update the screen, repeat. A while loop doing that simple dance, thousands of times per second.

for Loops: When You Know How Many Times

A for loop is a while loop with the counter built in: 

for (int i = 0; i < 10; i++) {
    printf("i = %d\n", i);
}

This prints 0 through 9. The three parts inside the parentheses are:

  1. int i = 0 — initialization. Set up the counter. Runs once before the loop starts.
  2. i < 10 — condition. Checked before each iteration. If false, stop.
  3. i++ — update. Runs after each iteration. Advance the counter.

Use for when you know how many times you want to loop. Use while when you're looping until some condition changes. They're interchangeable, but the right choice makes your intent clearer.

Why start at 0? In C, arrays (which we'll cover later) start at index 0. So programmers got into the habit of counting from 0. for (int i = 0; i < n; i++) is the most common loop in all of C. You'll write it in your sleep.

Logical Operators: Combining Conditions

Sometimes one condition isn't enough. You need "if this AND that" or "if this OR that": 

// AND: both must be true
if (age >= 18 && has_license) {
    printf("You can drive.\n");
}

// OR: at least one must be true
if (day == 6 || day == 7) {
    printf("It's the weekend!\n");
}

// NOT: flip true to false, false to true
if (!is_raining) {
    printf("No umbrella needed.\n");
}

You saw && in Crystal's event loop: ev.type < LASTEvent && handlers[ev.type]. Both conditions must be true — the event type must be valid AND there must be a handler registered for it. If either is false, we skip it.

break and continue

Two keywords that give you finer control over loops: 

// break: exit the loop immediately
for (int i = 0; i < 100; i++) {
    if (i == 42) break;   // stop at 42
    printf("%d\n", i);
}

// continue: skip the rest of this iteration, go to the next one
for (int i = 0; i < 10; i++) {
    if (i % 2 == 0) continue;  // skip even numbers
    printf("%d\n", i);              // only prints odd numbers
}

break is an emergency exit. continue is "skip this one and move on." Both are useful, both are common, and both make your loops more expressive.

Try It Yourself

  1. Write a program that prints the numbers 1 to 20, but says "fizz" for multiples of 3 and "buzz" for multiples of 5. If it's a multiple of both, print "fizzbuzz." (This is a classic programming exercise.)
  2. Write a while loop that counts down from 10 to 1, then prints "Liftoff!"
  3. Write a for loop that calculates the sum of all numbers from 1 to 100. (The answer should be 5050 — the same result Gauss figured out when he was seven years old.)
