Chapter 14: Advanced Topics — The Deep End

You're Ready for This

If you've made it through pointers, memory management, and debugging, you've already survived the hardest parts of C. The topics in this chapter are more advanced, but they build on everything you already know. Think of them as power tools — not required for every project, but incredibly useful when you need them.

Function Pointers: Functions as Data

Here's a mind-bending idea: in C, functions have addresses in memory, just like variables. That means you can store a pointer to a function and call it later. You can put function pointers in arrays, pass them to other functions, and even store them in structs. 

// Declare a function pointer type
typedef void (*EventHandler)(AuraWM *, XEvent *);

// Create an array of function pointers — one per event type
static EventHandler handlers[LASTEvent];

// Fill in the table during initialization
handlers[MapRequest]      = on_map_request;
handlers[UnmapNotify]     = on_unmap_notify;
handlers[ConfigureRequest] = on_configure_request;
handlers[ButtonPress]     = on_button_press;
handlers[KeyPress]        = on_key_press;

This is the actual event dispatch table from Crystal's events.c. Instead of a giant switch statement with a case for every event type, there's an array of function pointers. When an event arrives, the code looks up the handler: 

if (ev.type < LASTEvent && handlers[ev.type]) {
    handlers[ev.type](wm, &ev);   // call the function pointer!
}

handlers[ev.type] is a function pointer. Adding (wm, &ev) after it calls the function it points to. If the event type is MapRequest, this calls on_map_request(wm, &ev). If it's ButtonPress, this calls on_button_press(wm, &ev). One line of code handles all event types. That's the power of function pointers — data-driven dispatch instead of hard-coded branching.

This pattern is used everywhere in systems programming: plugin architectures, callback systems, signal handlers, virtual method tables. It's how C achieves polymorphism without classes or inheritance.

Bit Manipulation: Talking to Hardware

Computers think in bits. Sometimes you need to think in bits too — especially when working with hardware, protocols, or flags. C gives you operators that work at the bit level: 

&    // AND: both bits must be 1
|    // OR:  at least one bit must be 1
^    // XOR: exactly one bit must be 1
~    // NOT: flip all bits
<<   // left shift: multiply by 2
>>   // right shift: divide by 2

The most common use is flags — packing multiple boolean values into a single integer. X11 does this constantly: 

// X11 event mask — each bit enables a different type of event
long event_mask = SubstructureRedirectMask   // bit 20
               | SubstructureNotifyMask      // bit 19
               | StructureNotifyMask         // bit 17
               | PropertyChangeMask;         // bit 22

Each constant is a power of 2 (a single bit set). Using | (OR) combines them into one value with multiple bits set. The X server reads each bit to know which events you want. This is extremely memory-efficient — 32 boolean flags in a single 32-bit integer.

To check if a specific flag is set: 

if (e->state & MOD_KEY) {
    // The modifier key (Super/Command) is held down
}

& (AND) tests whether the MOD_KEY bit is set in the event's state field. If the result is non-zero, the modifier is active.

Preprocessor Macros: Code That Writes Code

We've seen #define for constants. But the preprocessor can do more — it can define macros that take arguments: 

#define MAX(a, b) ((a) > (b) ? (a) : (b))
#define MIN(a, b) ((a) < (b) ? (a) : (b))
#define CLAMP(x, lo, hi) (MIN(MAX(x, lo), hi))

These are textual substitutions. MAX(a, b) gets replaced with the ternary expression before compilation. The parentheses around every argument are critical — without them, operator precedence can produce wrong results. This is macro hygiene, and every C programmer learns it the hard way at least once.

Macros are also used for conditional compilation: 

#ifdef DEBUG
    fprintf(stderr, "[debug] x = %d\n", x);
#endif

If DEBUG is defined (via #define DEBUG or -DDEBUG on the command line), the debug print compiles in. Otherwise, it's stripped out entirely — zero runtime cost. This is how you add debug logging that doesn't slow down production builds.

The static Keyword: Three Different Meanings

The static keyword in C does three different things depending on context, which is frankly confusing. But each usage is important:

  • Static local variable: Retains its value between function calls. 
    void counter(void) {
    static int count = 0;  // initialized once, persists forever
    count++;
    printf("Called %d times\n", count);
}
  • Static function: Only visible within the current file. This is C's version of "private." 
    static void generate_shadow_texture(...) { ... }
// Only crystal.c can call this — it's invisible to other files
  • Static global variable: Same as static function — file-scope only. 
    static struct { ... } crystal;
// Only crystal.c can access this struct

Crystal makes heavy use of static for encapsulation. All the internal state and helper functions are static, keeping them hidden from the rest of the window manager. Only the functions declared in crystal.h are visible externally.

The volatile Keyword

One more keyword worth knowing: volatile tells the compiler "don't optimize reads/writes to this variable — its value can change at any time from outside the program." This is used when hardware registers, signal handlers, or other threads can modify a variable behind the compiler's back. You won't need it often, but when you do, nothing else will fix the bug.

Try It Yourself

  1. Create an array of function pointers: three functions that each print a different message. Store them in an array and call them in a loop.
  2. Write a #define SQUARE(x) ((x) * (x)) macro. Test it with SQUARE(5) and SQUARE(2 + 3). Then remove the parentheses and see what goes wrong with SQUARE(2 + 3).
  3. Use bit flags: define constants for READ = 1, WRITE = 2, EXECUTE = 4. Combine them with | and test them with &. This is exactly how Unix file permissions work.
