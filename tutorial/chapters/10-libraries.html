Chapter 10: Libraries — Standing on the Shoulders of Giants

Nobody Writes Everything From Scratch

Crystal Compositor is about 2,000 lines of C. But the functionality it uses — drawing to the screen, managing windows, rendering with OpenGL, loading PNG images, doing math — that's millions of lines of code. Crystal didn't write any of that. It uses libraries.

A library is a collection of pre-written, pre-compiled code that you link into your program. Instead of writing your own OpenGL renderer from scratch, you include <GL/gl.h> and call glBindTexture(). Instead of writing your own PNG decoder, you link against libpng and call png_read_image(). The library does the hard work. You just call its functions.

Crystal's Dependencies

Here are the libraries Crystal depends on, and what each one does:

  • Xlib (x11) — The fundamental X11 library. Creates windows, handles events, manages the display connection. This is how Crystal talks to the X server.
  • XComposite (xcomposite) — Redirects windows to off-screen pixmaps so Crystal can composite them with effects.
  • XDamage (xdamage) — Notifies Crystal when a window's contents change, so it only re-renders what's necessary.
  • XFixes (xfixes) — Miscellaneous X11 fixes and extensions, including cursor management.
  • OpenGL (gl) — GPU-accelerated rendering. Crystal draws all windows as textured quads using GL.
  • Cairo (cairo) — 2D vector graphics. Used for drawing title bars, text, and window decorations.
  • Pango (pangocairo) — Text layout and rendering. Draws window titles with proper font handling.
  • libpng (libpng) — Reads PNG image files. Used to load the traffic-light button assets.
  • libm (m) — The C math library. Provides sqrt(), sin(), cos(), exp(), etc.

That's a lot of libraries. Each one is a building block. Crystal orchestrates them all into a cohesive compositor. That's the art of systems programming — choosing the right libraries and wiring them together.

How Linking Works

When you compile a simple hello-world, the compiler handles everything: 

gcc hello.c -o hello

But when you use a library, you need to tell the compiler two things:

  1. Where the headers are (so the compiler knows about the functions): -I/path/to/headers
  2. Where the library files are and which ones to link: -L/path/to/libs -lname
# Compile with the math library
gcc circles.c -o circles -lm

# Compile with X11 and OpenGL
gcc compositor.c -o compositor -lX11 -lGL -lXcomposite

The -l flag says "link against this library." -lm links against libm (the math library). -lX11 links against libX11. The lib prefix and .so extension are implicit — you just give the short name.

pkg-config: The Library Finder

Managing all those -I and -l flags by hand is tedious and error-prone. Different systems install libraries in different places. That's where pkg-config comes in — it's a tool that knows where every library is installed and what flags you need. 

# Ask pkg-config what flags Cairo needs
$ pkg-config --cflags --libs cairo
-I/usr/include/cairo -lcairo

# Use it directly in your compile command
gcc myapp.c -o myapp $(pkg-config --cflags --libs cairo x11 gl)

pkg-config outputs the exact compiler flags needed for each library. You wrap it in $() so the shell substitutes its output into your compile command. Now you don't need to memorize where every library lives on every system. pkg-config handles it.

This is exactly what Crystal's build system (Meson) does behind the scenes. When meson.build says dependency('cairo'), it's calling pkg-config under the hood.

Static vs. Dynamic Libraries

Libraries come in two flavors:

  • Static libraries (.a files) — The library code is copied into your program at compile time. Your executable is self-contained but larger.
  • Dynamic/shared libraries (.so files on Linux, .dylib on Mac) — Your program loads the library code at runtime from a shared file. The executable is smaller, and multiple programs can share one copy of the library in memory.

Crystal uses dynamic linking for everything. When it calls glBindTexture(), the actual code for that function lives in libGL.so somewhere on your system. The operating system loads it into memory when Crystal starts up and connects the function calls to the right addresses. This is called dynamic linking, and it's how most software on Linux works.

Why dynamic linking? If ten programs all use OpenGL, they share one copy of libGL.so in memory instead of each embedding their own copy. That saves a lot of RAM. It also means you can update the library (fix a bug, patch a vulnerability) without recompiling every program that uses it.

Try It Yourself

  1. Write a program that uses the math library: #include <math.h>, call sqrt(2.0), and print the result. Compile with gcc math_test.c -o math_test -lm.
  2. If you're on Linux, try pkg-config --cflags --libs x11 to see what flags Xlib needs. Try it for cairo and gl too.
  3. Run ldd ./math_test (Linux) or otool -L ./math_test (Mac) to see which shared libraries your program depends on. You'll see libc, libm, and the dynamic linker itself.
