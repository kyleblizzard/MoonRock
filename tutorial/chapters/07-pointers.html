Chapter 7: Pointers — The Power and the Peril

Addresses, Not Values

Here's the concept that makes most people's eyes glaze over when they're learning C. But it doesn't have to be scary. Let me give you the one-sentence version:

A pointer is an address. It says WHERE something is, not WHAT it is.

Think of it like a street address. The address "742 Evergreen Terrace" isn't a house. It's a piece of paper that tells you where to find a house. A pointer works the same way. It doesn't contain data — it contains the memory address where data lives. 

int age = 28;          // a regular variable — holds the value 28
int *ptr = &age;       // a pointer — holds the ADDRESS of age

Two operators make pointers work:

  • & — the "address of" operator. &age means "give me the address where age lives in memory."
  • * — the "dereference" operator. *ptr means "go to this address and get what's there."
printf("Value of age: %d\n", age);      // 28
printf("Address of age: %p\n", &age);   // something like 0x7ffc9a3e4
printf("ptr points to: %d\n", *ptr);    // 28 (followed the address)

& says "give me the address of this thing." * says "go to this address and get what's there." They're inverses of each other. *(&age) is just age.

Why Pointers Exist

You might be thinking: "Why not just use the variable directly?" Fair question. Here are the two big reasons:

1. Functions can modify your data. In C, when you pass a variable to a function, the function gets a copy. Changes to the copy don't affect the original. But if you pass a pointer, the function can follow the address back to your original variable and modify it. 

// This does NOT work — the function gets a copy
void bad_double(int x) {
    x = x * 2;   // modifies the local copy, not the original
}

// This WORKS — the function gets the address
void good_double(int *x) {
    *x = *x * 2;  // follows the pointer, modifies the original
}

int n = 5;
bad_double(n);     // n is still 5
good_double(&n);  // n is now 10

2. Efficiency. The AuraWM struct is huge — it has an array of 256 Client structs inside it. If you passed it by value, C would copy the entire thing every time you called a function. That's thousands of bytes being duplicated for no reason. Passing a pointer is just passing an 8-byte address. Same access, zero waste.

Crystal's Pointers: The Real Deal

Now look at Crystal's function signatures with fresh eyes: 

bool crystal_init(AuraWM *wm);

AuraWM *wm is a pointer to an AuraWM struct. The function doesn't receive a copy of the entire window manager state — it receives the address of where that state lives. It can then read and modify any field of the window manager directly. When Crystal sets wm->running = false, it's modifying the actual running flag, not a copy.

And here's the X display connection: 

Display *dpy;

Display is an opaque struct defined by the X11 library. You never see what's inside it — you just hold a pointer to it and pass that pointer to X11 functions. This is a common pattern: the library manages the actual data, and you just carry around a pointer that says "here's my connection." It's like having a ticket number at a deli counter — you don't need to know how the kitchen works, you just hand them your number.

The Arrow Operator: ->

When you have a pointer to a struct, you access its fields with the arrow operator -> instead of the dot: 

Client *c = wm_find_client(wm, window_id);

// Using the arrow operator
c->focused = true;
c->x = 100;
c->y = 200;

// This is just shorthand for:
(*c).focused = true;   // dereference the pointer, then access the field

c->focused means "go to the struct that c points to, and access its focused field." It's shorthand for (*c).focused, which nobody wants to type. You'll see the arrow operator everywhere in Crystal's code — every function that takes AuraWM *wm uses wm-> to access the window manager's state.

NULL: The Pointer to Nowhere

NULL is a special value that means "this pointer doesn't point to anything." It's the absence of an address. You use it to indicate that something doesn't exist or hasn't been initialized. 

Client *wm_find_client(AuraWM *wm, Window w)
{
    for (int i = 0; i < wm->num_clients; i++) {
        if (wm->clients[i].client == w)
            return &wm->clients[i];
    }
    return NULL;   // window not found — return "nothing"
}

This function returns a pointer to a Client, or NULL if the window wasn't found. The caller must check for NULL before using the result: 

Client *c = wm_find_client(wm, w);
if (c) {
    // c is valid — safe to use
    printf("Found window: %s\n", c->title);
} else {
    // c is NULL — window doesn't exist
    printf("Window not found.\n");
}
Dereferencing NULL will crash your program. If you write c->title when c is NULL, your program tries to read memory at address 0, which the operating system does not allow. You'll get a "segmentation fault" — the most common crash in C programming. Always check for NULL before using a pointer.

Common Pointer Bugs

Pointers are powerful, but they come with traps. Here are the classics:

  • Dereferencing NULL: Using a pointer that's NULL. Your program crashes. Check before you dereference.
  • Use-after-free: Using a pointer to memory that has been freed (returned to the system). The memory might have been reused for something else. You'll get garbage data or a crash.
  • Dangling pointers: Returning a pointer to a local variable. When the function returns, that local variable is gone — the pointer now points to dead memory.
  • Uninitialized pointers: Declaring int *ptr; without setting it to anything. It could point anywhere. Dereferencing it is like driving to a random GPS coordinate and walking into whatever building you find.

These bugs are the reason C has a reputation for being dangerous. They're also the reason C is such an effective teacher. In languages with garbage collectors, you never think about where your data lives. In C, you have to. And that understanding makes you a fundamentally better programmer, no matter what language you use after.

Pointers are the reason C is both powerful and dangerous. Respect them. Every pointer should either point to something valid or be NULL. Every pointer should be checked before use. Every block of memory should have exactly one owner who is responsible for freeing it. Follow these rules and pointers become your greatest tool. Break them and they'll break you.
