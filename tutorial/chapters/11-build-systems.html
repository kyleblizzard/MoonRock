Chapter 11: Build Systems — The Recipe That Builds Your Program

The Problem Gets Real

Compiling one file is easy. Compiling twelve files with nine library dependencies, specific compiler flags, installation rules, and platform-specific tweaks? That's not something you want to type into the terminal by hand every time you make a change.

Crystal Compositor has 12 source files and depends on 9 libraries. The compile command would look something like: 

gcc src/main.c src/wm.c src/events.c src/client.c src/frame.c \
    src/decor.c src/assets.c src/ewmh.c src/input.c src/crystal.c \
    src/struts.c src/resize.c \
    $(pkg-config --cflags --libs x11 xcomposite xdamage xfixes xext \
      xrender xrandr cairo cairo-xlib pangocairo libpng gl) \
    -lm -o aura-wm

Nobody wants to type that. And that's a simple project. Real-world C projects have hundreds of files, conditional compilation, platform detection, test suites, and installation scripts. You need a build system.

What Is a Build System?

A build system is a program that reads a recipe file and runs all the compile and link commands for you. You describe what to build, and the build system figures out how to build it. The most common ones in the C world:

  • Make — The classic. Uses a Makefile. Old, powerful, and has a syntax that can be, uh, adventurous.
  • CMake — Generates Makefiles (or Ninja files, or IDE projects) from a CMakeLists.txt. Very widely used.
  • Meson — Modern, fast, and readable. Uses Python-like syntax. Generates Ninja build files. This is what AuraOS uses.

Crystal's meson.build

Here is the actual build file for the AuraOS window manager. Every line is doing something meaningful: 

project('aura-wm', 'c',
  version: '0.1.0',
  default_options: ['c_std=c11', 'warning_level=2'],
)

This declares the project: it's called aura-wm, it's written in C, version 0.1.0. It uses the C11 standard (not some ancient version of C) and warning level 2 (catch common mistakes but don't be overly paranoid). 

# Core X11 libraries
x11_dep       = dependency('x11')
xcomposite_dep = dependency('xcomposite')
xdamage_dep   = dependency('xdamage')
xfixes_dep    = dependency('xfixes')
xext_dep      = dependency('xext')
xrender_dep   = dependency('xrender')
xrandr_dep    = dependency('xrandr')

# Rendering libraries
cairo_dep     = dependency('cairo')
cairo_xlib_dep = dependency('cairo-xlib')
pangocairo_dep = dependency('pangocairo')
libpng_dep    = dependency('libpng')

# OpenGL
gl_dep        = dependency('gl')

Each dependency() call uses pkg-config behind the scenes to find the library. If a library is missing from your system, Meson will tell you immediately instead of failing with a cryptic linker error fifty lines deep. 

# The math library needs special handling — it's part of libc
# on some systems but separate on others
cc = meson.get_compiler('c')
math_dep = cc.find_library('m', required: true)

The math library (libm) doesn't have a .pc file for pkg-config, so we find it directly. Meson's compiler object knows how to search for it. 

aura_wm_sources = [
  'src/main.c',
  'src/wm.c',
  'src/events.c',
  'src/client.c',
  'src/frame.c',
  'src/decor.c',
  'src/assets.c',
  'src/ewmh.c',
  'src/input.c',
  'src/crystal.c',
  'src/struts.c',
  'src/resize.c',
]

executable('aura-wm',
  aura_wm_sources,
  dependencies: [
    x11_dep, xcomposite_dep, xdamage_dep, xfixes_dep,
    xext_dep, xrender_dep, xrandr_dep,
    cairo_dep, cairo_xlib_dep, pangocairo_dep, libpng_dep,
    gl_dep, math_dep,
  ],
  install: true,
)

This is the payoff. One executable() call that says: "Build a program called aura-wm from these 12 source files, linked against these 13 libraries, and install it to the system's bin directory." That's the entire build recipe. Clean, readable, and correct.

Building with Meson

Meson builds happen in two steps: 

# Step 1: Configure — create the build directory
meson setup build

# Step 2: Compile — actually build the program
meson compile -C build

meson setup build reads meson.build, checks that all dependencies are installed, and generates a build/ directory with Ninja build files. meson compile -C build runs the actual compilation. Ninja is extremely fast — it only recompiles files that have changed and runs compilations in parallel.

After making a change to one file, you just run meson compile -C build again. Ninja will recompile only the file you changed and re-link the binary. On a modern machine, that takes about a second.

Why Meson over Make? Makefiles are powerful but their syntax is full of gotchas (tabs vs. spaces, anyone?). Meson's syntax is cleaner, its error messages are friendlier, and it's significantly faster for large projects because it generates Ninja files. It also handles cross-compilation, subprojects, and platform detection out of the box. The trade-off is that Meson requires Python, while Make is available everywhere.

Try It Yourself

  1. Install Meson: pip install meson or sudo apt install meson ninja-build.
  2. Create a directory with a simple main.c (hello world) and a meson.build: 
    project('hello', 'c')
executable('hello', 'main.c')
  3. Build it: meson setup build && meson compile -C build
  4. Run it: ./build/hello
  5. Try adding a second source file and the math library dependency. Rerun meson compile -C build and see that it only rebuilds what changed.
