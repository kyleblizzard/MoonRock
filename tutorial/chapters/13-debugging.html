Chapter 13: Debugging — When Things Go Wrong (And They Will)

Bugs Are Inevitable

No one writes perfect code. Not beginners, not experts, not the people who wrote the C compiler. You will write bugs. The difference between a frustrated programmer and a productive one is how fast they find and fix them. Debugging is a skill, and like any skill, it gets better with practice.

C is particularly interesting to debug because it doesn't hold your hand. In Python, you get a nice traceback with line numbers and variable values. In C, you get "Segmentation fault (core dumped)" and nothing else. That's it. Figure it out.

But you have tools. Good tools. Let's walk through them.

The printf Debugger

The simplest debugging technique in history: print stuff. 

fprintf(stderr, "[debug] got here, x = %d\n", x);
fprintf(stderr, "[debug] pointer = %p\n", ptr);
fprintf(stderr, "[debug] about to call crystal_init()\n");

It's crude. It's effective. When your program crashes, sprinkle fprintf calls around the suspicious area and narrow down exactly where it dies. Crystal's code is full of these — every initialization step prints a status message: 

fprintf(stderr, "[crystal] Initializing Crystal Compositor...\n");
// ... lots of initialization code ...
fprintf(stderr, "[crystal] GLX context created successfully\n");
// ... more setup ...
fprintf(stderr, "[crystal] VSync enabled\n");

When Crystal fails to start, you look at the last message that printed. If you see "Initializing" but not "GLX context created," you know the problem is between those two points. Binary search for bugs. Narrow it down.

Use stderr, not stdout. stdout is buffered — your debug messages might not print before a crash. stderr is unbuffered (or line-buffered), so messages appear immediately. That matters when your program is about to segfault.

Compiler Warnings: Your First Line of Defense

The compiler catches a shocking number of bugs before your program ever runs. But only if you ask it to. Always compile with warnings enabled: 

# Bare minimum — catch the obvious stuff
gcc -Wall -Wextra myprogram.c -o myprogram

# Even more thorough — catch potential issues too
gcc -Wall -Wextra -Wpedantic -Wshadow myprogram.c -o myprogram

Common warnings that save your life:

  • "unused variable" — You declared something but never used it. Probably a typo.
  • "implicit declaration" — You called a function without including its header. The compiler is guessing what it looks like, and it's probably guessing wrong.
  • "format specifier mismatch" — You used %d to print a float, or %s to print an integer. This can crash your program or print garbage.
  • "comparison between signed and unsigned" — Subtle but dangerous. A signed -1 compared to an unsigned value can give the wrong answer.

Crystal's build system enables warning level 2 in Meson, which maps to -Wall -Wextra. Treat every warning as a potential bug. Fix them all. A clean compile with zero warnings is your baseline.

GDB: The Real Debugger

gdb (the GNU Debugger) is the power tool. It lets you pause your program at any point, inspect variables, step through code line by line, and examine the state of memory. It's like freeze-framing a movie and zooming in.

First, compile with debug symbols (-g): 

gcc -g -Wall myprogram.c -o myprogram

Then run it under gdb: 

gdb ./myprogram

Essential gdb commands: 

(gdb) run              # start the program
(gdb) break main       # set a breakpoint at the start of main
(gdb) break crystal.c:150  # break at line 150 of crystal.c
(gdb) next             # execute one line (step over function calls)
(gdb) step             # execute one line (step INTO function calls)
(gdb) print x          # show the value of variable x
(gdb) print *ptr       # dereference a pointer and show the value
(gdb) print wm->num_clients  # inspect a struct field
(gdb) backtrace        # show the call stack (which functions called which)
(gdb) continue         # resume running until the next breakpoint
(gdb) quit             # exit gdb

When your program segfaults inside gdb, it stops exactly at the crash site. Type backtrace to see the call stack — it tells you which function crashed and which function called it. Type print ptr to check if a pointer is NULL. Nine times out of ten, the bug is obvious once you can see the state at the crash point.

Common Bugs and How to Find Them

Segmentation Fault

The most common C crash. It means you tried to access memory that doesn't belong to you. Causes: dereferencing a NULL pointer, accessing an array out of bounds, using a freed pointer. 

// Classic segfault: forgot to check for NULL
Client *c = wm_find_client(wm, w);
c->focused = true;   // CRASH if c is NULL!

// Fix: check first
Client *c = wm_find_client(wm, w);
if (c) {
    c->focused = true;
}

Memory Leaks

Your program works but slowly eats all available RAM. Use Valgrind: 

valgrind --leak-check=full ./myprogram

Valgrind will tell you exactly which malloc call allocated memory that was never freed, with the file name and line number. It's magic.

Off-by-One Errors

Accessing array[n] when the valid range is 0 to n-1. Or looping i <= n when you meant i < n. These are subtle and common. Double-check your loop bounds every time.

Uninitialized Variables

Using a variable before assigning it a value. The variable contains whatever garbage was in that memory. This can cause "works on my machine but not on yours" bugs, because the garbage is different every time. 

// Bug: result is uninitialized if the loop doesn't run
int result;
for (int i = 0; i < count; i++) {
    result = compute(i);
}
printf("%d\n", result);   // garbage if count == 0!

Fix: int result = 0; Always initialize.

Try It Yourself

  1. Write a program with an intentional bug (access an array out of bounds). Compile with -g, run it under gdb, and use backtrace to find the crash location.
  2. Write a program with a memory leak (malloc without free). If you have Valgrind installed, run it through and read the leak report.
  3. Compile a program with -Wall -Wextra and fix every warning. Notice how many potential bugs the compiler catches for you.
