Chapter 6: Structs — Building Your Own Types

When One Variable Isn't Enough

Let's say you're writing a program that deals with points on a screen. A point has an X coordinate and a Y coordinate. You could use two separate variables: 

int point_x = 100;
int point_y = 200;

That works for one point. But what about ten points? Twenty? Now you need point1_x, point1_y, point2_x, point2_y... it gets ugly fast. And if you want to pass a point to a function, you have to pass two separate arguments every time. It's clumsy. It doesn't scale.

What you really want is to say: "A point IS an x and a y, bundled together as one thing." That's exactly what a struct does.

Your First Struct

struct Point {
    int x;
    int y;
};

You just created a new type. struct Point is now a type, just like int or float. It contains two integers: x and y. The individual pieces inside are called fields or members.

To use it: 

struct Point cursor;
cursor.x = 100;
cursor.y = 200;

printf("Cursor at (%d, %d)\n", cursor.x, cursor.y);

The dot (.) is how you access a field inside a struct. cursor.x means "the x field of the cursor variable." It reads naturally — almost like English.

You can also initialize a struct all at once: 

struct Point cursor = { 100, 200 };   // x = 100, y = 200

This is the moment where C stops feeling like math homework and starts feeling like building something. Structs let you model the world. A window has a position, a size, a title. A player has health, a name, a score. A pixel has red, green, blue values. You define the shape of your data, and then you work with it as a single, coherent unit.

Crystal's WindowTexture: A Real Struct

Here's an actual struct from Crystal Compositor. This is how Crystal tracks each window that's being composited on screen: 

struct WindowTexture {
    Window  xwin;        // The X11 window ID
    Pixmap  pixmap;      // Off-screen image from XComposite
    GLuint  texture;     // OpenGL texture handle (GPU-side)
    Damage  damage;      // Notifies us when contents change
    int     x, y, w, h; // Position and size on screen
    bool    dirty;       // Needs its texture refreshed?
    bool    has_alpha;   // Does this window have transparency?
    bool    bound;       // Is the texture currently bound?
    GLuint  shadow_tex;  // Pre-computed shadow texture
    int     shadow_tex_w, shadow_tex_h;  // Shadow dimensions
};

Look at all the data bundled together. A window on your screen isn't just one thing — it has a position (x, y), a size (w, h), a GPU texture for its contents, a shadow texture, flags for whether it's transparent or needs updating... All of this lives together in one struct because it all describes the same window.

Without structs, you'd need separate arrays for each piece of data: window_x[256], window_y[256], window_w[256], window_h[256], window_dirty[256]... Nightmare fuel. With a struct, it's clean: 

struct WindowTexture windows[256];

One line. An array of 256 window textures. Each one is a complete package of everything Crystal needs to know about that window. To access the width of window number 5: 

int width = windows[5].w;

Simple. Readable. Correct.

Crystal's Client Struct: Modeling a Managed Window

The window manager has its own struct for tracking windows, separate from Crystal's: 

struct Client {
    Window client;       // The application's actual window
    Window frame;        // Our frame window (wraps the client)
    int    x, y;         // Frame position on screen
    int    w, h;         // Client content size
    bool   mapped;       // Is this window visible?
    bool   focused;      // Is this the active window?
    char   title[256];   // Window title text
    bool   zoomed;       // Is this window maximized?
    int    saved_x, saved_y, saved_w, saved_h;  // Pre-zoom size
};

Every window you see on your desktop has a Client struct behind it. When you click a window to focus it, the window manager sets c->focused = true. When you drag a window, it updates c->x and c->y. When you maximize a window, it saves the current size into saved_x/y/w/h, sets zoomed = true, and resizes the window to fill the screen. When you un-maximize, it reads those saved values back.

Notice char title[256] — that's a character array (a string) embedded right inside the struct. It can hold up to 255 characters of window title text (plus a null terminator, which we'll talk about when we cover strings). The title "Firefox" sits right there in the struct, alongside the position and size data.

The Global State: Struct of Structs

Here's where it gets really satisfying. The window manager's entire state is one big struct: 

struct AuraWM {
    Display *dpy;           // Connection to the X server
    int      screen;         // Which screen we're on
    Window   root;           // The root (desktop) window
    int      root_w, root_h; // Screen dimensions

    Client   clients[256];   // All managed windows
    int      num_clients;    // How many windows we're tracking
    Client  *focused;        // Currently focused window

    bool     running;        // Is the WM still running?
    bool     dragging;       // Is the user dragging a window?
    Client  *drag_client;    // Which window is being dragged
};

See that? clients[256] — an array of 256 Client structs. A struct inside a struct. The AuraWM struct contains everything: the display connection, the list of windows, the current focus state, drag state... the entire window manager's brain, in one variable.

In main(), this entire world is created with one line: 

static AuraWM wm;

One variable. It holds everything. And then every function that needs access to the window manager's state receives a pointer to this struct: crystal_init(&wm), events_run(&wm), wm_shutdown(&wm). That's how state flows through a C program — you create it in one place and pass references to it everywhere else.

Structs are how C programmers think about design. Before writing any code, experienced C programmers ask: "What data do I need? What belongs together?" They design their structs first, then write functions that operate on them. The struct is the design. If your structs make sense, the code practically writes itself.

Structs and Functions

Structs and functions go hand in hand. You define a struct to represent something, then write functions that operate on it. Here's a small self-contained example: 

#include <stdio.h>
#include <math.h>

struct Point {
    float x;
    float y;
};

// Calculate the distance between two points
float distance(struct Point a, struct Point b)
{
    float dx = b.x - a.x;
    float dy = b.y - a.y;
    return sqrt(dx * dx + dy * dy);
}

int main(void)
{
    struct Point origin = { 0.0, 0.0 };
    struct Point target = { 3.0, 4.0 };

    printf("Distance: %.2f\n", distance(origin, target));  // 5.00
    return 0;
}

The distance function takes two Point structs and returns the distance between them. Clean, readable, reusable. You could call this function a thousand times with different points and it would just work.

typedef shortcut: Tired of typing struct Point everywhere? C has a keyword called typedef that creates an alias:

typedef struct { int x, y; } Point;

Now you can just write Point p; instead of struct Point p;. Crystal's code uses typedef for AuraWM and Client, which is why you see AuraWM *wm instead of struct AuraWM *wm in function signatures.

What's Next

You've now got the core building blocks: variables to hold data, functions to organize behavior, control flow to make decisions, and structs to model complex things. These four concepts are the foundation of everything in C.

In the next set of chapters, we'll tackle the topics that separate C from friendlier languages: pointers, memory management, arrays, and strings. That's where C gets its reputation — and where it gets its power. You'll understand what that * means in AuraWM *wm, why we pass &wm to functions, and how Crystal manages to track 256 windows worth of GPU textures without breaking a sweat.

But for now, take a breath. You know more C than you think. Go back and look at Crystal's main() from Chapter 2. I bet it makes a lot more sense now than it did then.
